    Overview

    API Endpoint

    https://ovloop.io

    OVLoop API is a simple RESTful interface that enables third-party components to access functionality of the OVLoop one-touch loyalty platform. This document is the official reference for that functionality.

    This document should be used by third-party developers who wish to integrate their products with OVLoop.

    We have language bindings in Shell (other languages to be provided in the future)! You can view code examples in the dark area to the right, and you can switch the programming language of the examples with the tabs in the top right.

    OVLoop API is organized around REST and uses HTTP verbs and response codes that most clients are familiar with. All requests must be formatted as Content-Type: application/json (except where noted). All responses are also formatted JSON, encoded in the UTF-8 character encoding (Content-Type: application/json; charset=utf-8). The Accept header will be ignored for all requests.

    Security

    OVLoop uses JSON Web Tokens with OAuth2 for authentication of clients (i.e. "profiles") with OVLoop API. JSON Web Tokens are an open, industry standard [RFC 7519] https://tools.ietf.org/html/rfc7519) method for representing claims securely between two parties. Tokens received are used within headers of subsequent requests to the OVLoop API.

    1. First, encode your client credentials as instructed in the Authentication section below.
    2. Then you exchange these client credentials for an OAuth 2.0 JSON Web Bearer token.
    3. Once you’ve done that, you simply include your token with all of your OVLoop requests.

    An authorization header containing a valid access token must be included in every OVLoop API request like follows:

    Authorization: bearer {{token}}

    You must replace {{token}} with your JSON web token.

    The bearer token is your key to accessing all of your resources via OVLoop API for the next 24-hour period. Keep it safe.

    SSL must be used in all OVLoop API interactions.

    API Update Guidelines

    Our API is constantly being updated with new features and improvements to existing ones. These updates ensure the API evolves with customer and industry demands. Managing this change to ensure we provide a reliable, consistent, and backward compatible API design is a critical challenge for our team. Every new and improved feature is put through a peer review process to ensure:

    1. it will not change an existing resource to the extent it breaks current customer implementations,
    2. that it is consistent with existing resources (therefore predictable and easy to use).

    The following guidelines are used in our peer review process to understand what constitutes acceptable change for our API and our customers.

    Changes we avoid:

    • Removing or renaming request or response content fields
    • Removing or renaming query string parameters
    • Removing or changing the location of a resource
    • Changing the API domain

    Changes we approve and introduce:

    • Adding new resources
    • Adding new, optional request content fields
    • Adding new, optional query string parameters
    • Adding new response content fields
    • Adding new response header information

    If, for any reason, we cannot avoid any of the non-backward compatible changes mentioned above, we will implement a deprecation schedule. This schedule will be provided to all customers with the goal of giving them adequate time to accommodate the updates.

    Looking further out, as our API evolves and industry standards dictate, we do anticipate the possibility that an entirely new API version will be needed. If and when that happens we will, once again, provide a deprecation schedule for the existing version that will ensure all customers have time to upgrade.

    Additional information:

    Undocumented resources and properties are considered to be in beta release and are subject to change without notification.

    OVLoop REST interface

    OVLoop API REST interface is powered by .NET Core Web API. It exposes server functionality to the clients via a combination of HTTP and XML or JSON protocols.

    JSON

    OVLoop API supports the JSON language for the REST protocol via content negotiation mechanism. The same values should be used for Content-Type HTTP header when submitting data in the request body (POST and PUT methods).

    For authentication, only 'x-www-form-urlencoded' format is accepted.

    Allowed HTTP Requests

    • GET - Get a resource or list of resources.
    • POST - To create a resource.
    • PUT - To update a resource.

    Errors

    OVLoop API always returns relevant HTTP status code which includes the errorCode (application specific code). The error result can also specify one or more parameter elements, which further specify what went wrong.

    An error response with an http code 404 (Bad Request):

    HTTP/1.1 404 Bad Request: { "succeeded" : false, "error" : {{errorMessage}} }

    A successful GET response:

    HTTP/1.1 200 OK

    A successful PUT response:

    HTTP/1.1 201 OK

    When an error is encountered you will receive an HTTP status code along with a message and error code in the body of the response. The message is intended to give a user-friendly explanation of the error while the error codes are designed to be machine readable codes that applications can use to better understand the context of the error and react appropriately.

    We use the following status codes for errors:

    Status Code Meaning
    400 Bad Request - The request could not be understood or was missing required parameters.
    401 Unauthorized - The authentication process failed, or the access token is not valid.
    403 Forbidden -- Access to the requested endpoint is restricted for the given caller.
    404 Not Found -- The specified resource could not be found.
    405 Method Not Allowed -- You tried to access a resource with an invalid method.
    406 Not Acceptable -- You requested a format that isn't json.
    410 Gone -- The requested resource entity has been removed from our servers.
    429 Too Many Requests -- Too many requests within a specific time limit.
    500 Internal Server Error -- We had a problem with our server. Try again later.
    503 Service Unavailable -- We're temporarily offline for maintenance. Please try again later.

    Authentication

    OVLoop uses JSON Web Tokens with OAuth2 for authentication of clients (i.e. "profiles") with OVLoop API. JSON Web Tokens are an open, industry standard [RFC 7519] https://tools.ietf.org/html/rfc7519) method for representing claims securely between two parties. Tokens received are used within headers of subsequent requests to the OVLoop API.

    The following properties are used with Authentication:

    Available Authentication data (property names)

    Property Type Required Description
    client_id string Y The client ID. This ID will be provided to you by the OVLoop support team and is application specific.
    grant_type string Y The grant type. Use "client_credentials" for accesing resources from a server.
    client_secret GUID Y The client secret provided to you when being issues the client Id.

    Generate Access Token

    JSON web tokens tokens are long-lived (24 hours) authentication tokens that can be used to replace expired access tokens without providing user credentials. JSON web tokens are issued when an access token is requested using OVLoop client credentials.

    When requesting an access token using a refresh token you must use a grant type of ‘client_credentials’. The client Id and client secret must also be provided in the request header.

    HTTP Request

    POST https://ovloop.io/api/oauth2/token

    Request Headers

    Content-Type: application/x-www-form-urlencoded

    To authorize, use this code:

    # With shell, you can just pass the correct header with each request.
# Make sure to replace `{{clientId}}` with your client Id and `{{clientSecret}}` with your client secret.

curl -i -X POST
  -H "Content-Type: application/x-www-form-urlencoded"
  -d "client_id={{clientId}}&grant_type=client_credentials&client_secret={{clientSecret}}"
  --url "https://ovloop.io/api/oauth2/token"

    Sample Response Body

    {
    "token_type": "Bearer",
    "access_token": "{{token}}",
    "expires_in": 86400
}

    Response Body

    Property Description
    token_type The token type.
    access_token The access token. Use this token to authorize other API calls by placing it in the headers such as Authorization: bearer {{access_token}}
    expires_in The number of seconds from token issuance to expiration.

    Response Status Codes

    Status Code Meaning
    200 successful request
    404 if a validation fails
    500 unknown server error

    Application Error Codes

    Status Code Meaning
    500 unknown server error
    The returned value in `{{token}}` is the JSON web token to be used with all subsequent API requests.

    Organization

    The Organization resource represents an organization boarded on the OVLoop platform that can have one or many merchant listings/destinations with offers or loyalty rewards programs.

    The following properties are used with Organization:

    Available Authentication data (property names)

    Property Type Required Description
    organizationId GUID Y A system-generated GUID that uniquely identifies the organization
    name string Y The name of the organization.
    parentOrganizationId GUID Y The GUID of the parent organization.

    Create a child organization

    This endpoint creates a child organization underneath an existing parent (e.g., a reseller or partner).

    The only field needed to create a child organization is the organization name. The organizationId and parentOrganizationId are populated by the system.

    A successful create operation will return a clientId and clientSecret. These values should be saved by the caller and used to generate authorization tokens for acessing other API methods that create, edit, or retrieve resources belonging to this organization.

    In addition, the organizationId will be returned in the Location header and should be saved to update the organization after creation.

    HTTP Request

    POST https://ovloop.io/api/organization

    Request Headers

    Content-Type: application/json

    Authorization: bearer {{token}}

    To create a sample child organization, use this code:

    # With shell, you can just pass the correct header with each request.
# Make sure to replace `{{token}}` with your current valid JSON web token (see Authentication).

curl -i -X POST
  -H "Content-Type: application/json"
  -H "Authorization: bearer {{token}}"
  -d "{ 'name': 'Sample Organization' }"
  --url "https://ovloop.io/api/Organization/Child"

    Sample Request Body

    {
    "name" : "Sample Organization"
}

    Sample Response Body

    {
    "clientId": "a1e85778985f4555bac159c8e8e54d8d",
    "clientSecret": "lGF-hponqH-UAVBLSfHpOavArNNV7_6K4i8p8h1TTxg",
    "allowRestrictedApiGrant": false,
    "allowProfileGrant": false,
    "allowClientCredentialsGrant": true,
    "applicationId": null,
    "organizationId": "a1646e2d-1caf-4173-bab0-88b6c7893651"
}

    Request Body

    Property Type Required Description
    name string Y The name of the organization.

    Response Body

    Property Description
    clientId The client_id value used for authentication for creating resources that belong to this organization.
    clientSecret The client_secret value used for authentication for creating resources that belong to this organization.
    allowRestrictedApiGrant Specifies whether the created child organization has restricted API grant permissions. Default is false.
    allowProfileGrant Specifies whether the created child organization has profile grant permissions. Default is false.
    allowClientCredentialsGrant Specifies whether the created child organization has client credentials grant permissions. Default is false.
    applicationId ApplicationId for the primary application associated with the organization. Default is null.
    organizationId The organization Id for the created child organization.
    The caller should retain the `organizationId` generated within their own system for future API calls for the specific organization.

    Response Status Codes

    Status Code Meaning
    200 successful request
    401 the client is not properly authenticated
    404 if a validation fails
    500 unknown server error

    Application Error Codes

    Status Code Meaning
    500 unknown server error

    Listing

    The Listing resource represents the merchant location(s) which is sponsoring the offer(s) and/or loyalty program(s).

    The following properties are used with Listing:

    Available Listing data (property names)

    Property Type Required Description
    title string Y The listing title screen seen by consumers inside the OVLoop app.
    listingName string Y The listing name seen in a merchant admin portal. Not seen by consumers.
    description string Y A description of the business; seen by consumers inside the OVLoop app.
    address string Y The listing's address.
    phoneNumber string Y The listing's phone number without any formatting or special characters.
    categoriesList array (Category) Y TO DOCUMENT (COPY FROM EXAMPLE FOR NOW)
    weblink string N The URL of the listing's website; seen by consumers inside the OVLoop app.
    facebookUrl string N The listing's Fackbook URL; seen by consumers inside the OVLoop app.
    twitterUrl string N The listing's Twitter URL; seen by consumers inside the OVLoop app.
    latitude number Y The latitude of the listing's location. This is used for notifications triggered by a geofence around the listing's location.
    longitude number Y The longitude of the listing's location. This is used for notifications triggered by a geofence around the listing's location.
    radius number Y The radius of the geofence for the listing. Allowed values are 200 or 500.
    notificationTitle string Y The body of the notification triggered by the geofence.
    rapidAppIdList array Y TO DOCUMENT (COPY FROM EXAMPLE FOR NOW)
    scheduleDays array (ScheduleDay) Y The listing's business hours. Please see the ScheduleDay properties table below for the nested object's format.

    Available ScheduleDay data (property names)

    Property Type Required Description
    dayOfWeek number Y The day of week, starting from Sunday, represented as a number between 0 and 6.
    startTime string Y The start time of this period (24-hour format HH:MM:SS).
    endTime string Y The end time of this period (24-hour format HH:MM:SS).
    closed boolean Y Signifies if the listing is closed for the specified day. The default is false.
